Why Goblint?


Developing software that is free of bugs is a complex and challenging task, and even more so in a multi-threaded setting. Goblint supports developers in this endeavor, by providing insights into the code and flagging potential bugs. This flagging of bugs goes way beyond what inters typically can provide: Instead Goblint carefully considers the semantics of the entire program to flag all the bugs that could occur.

Goblint builds on over a decade's worth of development and research in field of Static Analysis by Abstract Interpretation, leveraging the expertise of the contributing research groups at the University of Tartu and at Technical University of Munich on Abstract Interpretation, the semantics of the C programming language, Compiler Engineering, and Efficient Fixpoint Computation to create an analyzer targeting real-world applications.

It also serves as the proving ground of our research ideas, so is constantly being enhanced with new and exciting ideas.

After over a decade of engineering, Goblint is a robust framework that can handle real-world applications and not just toy examples.

Highlights


Sound by construction

Goblint is bases on a strong theoretical foundation, namely the concept of Abstract Interpretation as proposed by Patrick and Radhia Cousot. Thus, when Goblint claims a program is free of (a given class of) bugs, the program actually is!

Path-sensitivity

Goblint is path-sensitive, so it deals with conditional locking schemes as well as possibly failing, interruptible and probing locking primitives. Mutexes may be handled both path-sensitively and symbolically. This is a key ingredient in reducing the number of false positives.

Thread-modularity

Goblint performs thread-modular analysis of C programs. It computes a sound upper approximation of all possible thread interleavings using global invariants for shared data, while avoiding the combinatorial explosion of considering all interleavings.

Incrementality

Instead of starting analysis from-scratch after a modification, Goblint is able to reuse its results for a previous version of the program and update them to account for the effect of changes in a fraction of the time a from-scratch analysis takes.




Supported Properties


Race detection

Goblint started its life as a framework for race detection, and still excells here. Goblint checks that all accesses to shared memory are protected by a common mutex, also taking into account which threads can execute in parallel, and which can't.

Reach safety

Goblint is capable of checking whether some assert is guaranteed to hold at a given program point, or whether a certain program is unreachable. This can be used to encode almost any safety property of interest.

Integer Overflows

Goblint can verify that programs are free of overflows of (signed) overflows of integers, which are notorious causes of Undefined Behavior and bugs in real-world programs.

Goblint for...


... software developers.

Goblint is free to try out and comes with both a command line interface, the possibility to integrate it into your CI workflow, and an easy to use IDE integration. If your program is open source, feel free to contribute it to our benchmark suite. We're happy to help if there are issues. Give it a try!

... students.

We are continuously looking for motivated students who are excited to learn about Static Analysis by Abstract Interpretation and want to contribute to Goblint. This can be in a course, as a HiWi, or as a Bachelor's or Master's thesis, both at UTartu and at TUM. Just reach out to any of us if you are interested!

... researchers.

Goblint is both a mature framework and easy to extend. As such, it is well-suited to prototype your own ideas without having to implement all the standard things from scratch. Also, building on Goblint makes it easy to test your ideas on real-world applications instead of being limited by engineering issues!

Architecture


Goblint is implemented in OCaml and uses a fork of CIL that has been extended with support for C99 and (most of C11) as its parser frontend for the C language. The architecture of Goblint is designed to be modular. Analyses, which are defined by their abstract domains and transfer functions, can be activated via runtime configuration options. A flexible query system allows for communication between analyses. Together, the combined analyses and the control-flow graphs of the functions in the program provide the side-effecting constraint system, which is solved by some local generic solver. Post-processing the solution yields results for the analysis.
